# 蓝领智汇平台合同管理需求分析报告

## 一、概述

### 1.1 项目背景

蓝领智汇平台是一款面向劳务公司、工厂和工人的一体化SaaS管理系统，旨在实现蓝领人力资源的全流程数字化管理。合同管理作为平台的核心业务模块之一，承担着连接工人入职工序与在职管理的重要职责，是蓝领劳务关系确立的关键环节。根据平台整体规划，合同管理模块位于一级菜单中的第五个位置，充分体现了其在业务流程中的核心地位。

在蓝领劳务行业中，合同管理面临着诸多特殊挑战。首先，工人流动性大，合同签订频繁，需要高效的处理流程；其次，工人主要通过移动端进行操作，而劳务公司管理员则通过PC端进行管理，需要实现多端协同；再者，合同涉及多方主体，包括劳务公司、工厂和工人，需要清晰的关系映射；最后，合同状态变更需要与工人的生命周期状态联动，如合同签订后工人状态应从“面试邀约”变更为“进厂”。

本报告基于pc-feature-spec.md和ui-design-spec.md两份规范文档，结合现有代码实现，对合同管理功能进行全面的需求分析，旨在为后续开发提供清晰的功能指引和设计依据。报告涵盖功能模块、业务流程、数据权限、模块关联、界面设计和审批流程等维度，力求完整呈现合同管理模块的业务需求和技术要求。

### 1.2 分析目标

本需求分析报告的核心目标是明确合同管理模块的功能边界和业务规则，具体包括以下几个方面：第一，梳理合同管理的功能模块结构，识别需要开发的功能点；第二，描述合同管理的核心业务流程，包括合同签订、状态变更等关键环节；第三，明确合同管理的数据权限配置方案，确保不同角色能够按照权限查看和操作合同；第四，分析合同管理与其他业务模块的关联关系，确保数据流转的连贯性；第五，提出界面设计和用户体验的具体要求；第六，配置审批流程相关参数。

通过本报告的分析结果，开发团队可以准确理解合同管理模块的业务需求，UI设计师可以获取界面设计的技术约束，产品经理可以依据报告进行进度规划和资源调配。报告同时为后续的数据库设计、API设计和前端开发提供参考依据，确保各环节工作的一致性和协调性。

## 二、功能模块分析

### 2.1 模块定位

合同管理是一级菜单“劳务公司端”的核心功能模块，主要服务于劳务公司的管理人员和正式员工。根据平台业务架构，合同管理的主要职责包括：管理劳务公司与工人之间的劳务合同关系，记录合同的关键信息如甲方（劳务公司）、乙方（工人）、结算方式、合同期限等；跟踪合同的状态流转，从“未签订”到“签订中”再到“已签订”或“已取消”；管理与合同相关的附件文件，包括合同文本、工人证件等；为结算管理模块提供合同金额等数据支持。

从业务视角来看，合同管理模块是工人进入工厂工作的前提条件。只有完成合同签订的工人才会进入“在职管理”阶段，因此合同管理的准确性和及时性直接影响后续的业务流程。同时，合同管理也是结算管理的重要数据来源，合同中约定的结算方式（月结、日结、周结）直接决定了工资结算的方式。

### 2.2 签订合同功能

签订合同是合同管理模块的核心功能，承载着合同创建、编辑、查看、删除等基础操作。根据pc-feature-spec.md的规范要求，该功能需要实现以下具体能力：

**列表展示功能**：合同列表页面需要展示所有当前登录账号负责的合同信息，支持按照数据权限进行数据过滤。列表展示的字段包括：合同编号、甲方（默认账号所属企业，可修复）、乙方（从工人信息中进行选择，需要是初始通过的工人）、结算方式（日结、月结、周结）、合同状态（未签订、签订中、已签订、已取消）、合同签订日期、合同生效日期、合同过期日期、合同金额。列表需要支持排序功能，包括按合同编号、签订日期、生效日期、过期日期、金额等字段进行升序或降序排列。同时需要支持按合同状态、结算方式、签订日期范围进行筛选搜索。

**新增合同功能**：点击“新增合同“按钮跳转到合同表单页面。表单需要包含以下字段：合同编号（系统自动生成或手动输入）、甲方（默认带出当前登录账户所属企业名称，可编辑）、乙方（通过人员选择器从工人信息中选择，需要是已完成面试通过的工人）、结算方式（支持月结、日结、周结三种选项）、合同状态（默认为“未签订”）、合同签订日期、合同生效日期、合同过期日期、合同金额、合同内容（富文本编辑器，支持图文混排）、合同附件（文件上传组件，支持多文件上传）。

**编辑合同功能**：在合同列表中，点击“编辑“按钮可跳转到合同编辑页面。编辑页面的表单结构与新增页面一致，需要回显已有数据。编辑功能仅在合同状态为“未签订”或“签订中“时可用，已签订或已取消的合同不允许编辑。

**删除合同功能**：在合同列表中，点击“删除“按钮会弹出确认对话框，进行二次确认后执行删除操作。删除功能同样仅在合同状态为“未签订”或“签订中“时可用。删除操作是物理删除，执行后数据不可恢复。

**查看详情功能**：在合同列表中，点击“查看“按钮可跳转到合同详情页面。详情页面展示合同的所有字段信息，包括基本信息卡片（合同编号、甲方、乙方、结算方式、状态、签订日期、生效日期、过期日期、金额）、合同内容卡片（富文本格式的合同正文）、合同附件卡片（附件列表，支持预览和下载）、操作记录卡片（创建、修改、状态变更等操作的日志）。

### 2.3 辅助功能

除核心的签订合同功能外，合同管理模块还需要支持以下辅助功能，以提升用户体验和业务效率：

**合同打印功能**：在合同详情页面，提供“打印“按钮，支持调用浏览器打印功能或系统打印模板进行合同打印。打印功能需要预先配置打印模板，确保打印格式的规范性和美观性。打印模板的管理由系统管理模块的“打印管理“功能负责。

**合同导出功能**：支持将合同列表数据导出为Excel文件，导出内容应包含列表展示的所有字段。导出功能支持按当前筛选条件进行过滤，即只导出符合搜索条件的合同记录。

**合同模板功能**：提供常用合同模板的创建和管理功能，模板中可以预定义合同内容结构。新增合同时可以选择使用模板，快速填充合同内容，提升工作效率。

**合同续签功能**：当合同即将到期时，系统应能够提醒用户进行合同续签。续签功能实际上是创建新合同，但会自动带出原合同的部分信息如甲方、乙方、结算方式等，简化用户操作。

**批量操作功能**：支持批量选择合同进行批量删除或批量导出。批量操作需要在列表页面提供复选框，勾选需要操作的记录后，点击批量操作按钮执行相应操作。

## 三、业务流程分析

### 3.1 合同签订流程

合同签订是合同管理模块的核心业务流程，贯穿了从合同创建到合同生效的完整生命周期。根据业务场景分析，合同签订流程可以分为以下几个阶段：

**第一阶段：合同创建**。劳务公司操作员点击“新增合同“按钮，进入合同表单页面。操作员填写合同基本信息，包括合同编号（系统自动生成）、甲方（当前企业）、乙方（从工人列表中选择）、结算方式、合同签订日期、合同生效日期、合同过期日期、合同金额等。操作员可以通过富文本编辑器编辑合同正文内容，也可以上传合同附件如合同扫描件、工人证件复印件等。填写完成后，点击“保存草稿“或”提交“按钮。保存草稿将合同保存为草稿状态，提交则进入下一步审核流程。

**第二阶段：合同审核**（可选）。如果系统配置了合同审批流程，提交的合同需要经过相关人员审核。审核人员查看合同内容，确认无误后点击“审核通过“；如有疑问可填写驳回原因后点击”审核驳回“。审核通过后，合同状态变更为“签订中”；审核驳回后，合同状态保持不变，操作员可以根据驳回原因修改后重新提交。

**第三阶段：合同签订完成**。对于无需审核或审核通过的合同，操作员可以将合同状态变更为“已签订”，表示合同签订流程已完成。合同状态变更为“已签订”后，合同正式生效，系统应自动触发后续流程，如更新工人的生命周期状态为“进厂”，通知相关人员等。

**第四阶段：合同履行与终止**。合同签订完成后进入履行期，期间可以查看合同详情、下载合同附件等操作。当合同到期或因其他原因需要终止时，操作员可以将合同状态变更为“已取消”。已取消的合同不允许再进行修改。

### 3.2 状态流转规则

合同状态是合同管理的重要属性，定义了合同在生命周期中所处的阶段。根据平台规范，合同状态分为以下四种：

**未签订（UNSIGNED）**：表示合同刚刚创建，还未完成签订流程。这是合同的初始状态，在此状态下可以对合同进行编辑、删除操作，也可以提交进行审核或直接变更为“签订中”。

**签订中（SIGNING）**：表示合同正在签订过程中，可能正在进行审核或等待工人确认。在此状态下可以对合同进行编辑、删除操作，也可以完成签订将状态变更为“已签订”，或取消签订将状态变更为“已取消”。

**已签订（SIGNED）**：表示合同已完成签订流程，正式生效。在此状态下合同不允许编辑和删除，只能查看详情或进行打印操作。合同过期后系统会自动将状态变更为“已取消”，或由操作员手动变更。

**已取消（CANCELLED）**：表示合同已被取消，可能是因为审核驳回、合同到期、双方协商终止等原因。在此状态下合同不允许编辑和删除，只能查看详情。

状态流转规则可以总结如下：从“未签订“可以流转到”签订中“或”已取消“；从”签订中“可以流转到”已签订“或”已取消“；从”已签订“可以流转到”已取消“；”已取消“为终止状态，不可再变更。其他状态流转如从“已签订“回到”签订中“或从”已取消“回到”未签订“等均不允许。

### 3.3 与工人生命周期的联动

合同管理模块与工人生命周期状态紧密相关，两者需要实现联动。当合同状态变更为“已签订“时，系统应自动将对应工人的生命周期状态从“面试邀约“或”工厂面试“变更为”进厂“。这一联动机制确保了工人状态的实时性，避免了人工操作带来的延迟和错误。

反过来，当工人状态变更为“离职“时，对应的合同状态也应该相应更新。如果合同还未到期，系统可以自动将合同状态变更为”已取消“，并记录变更原因。这一联动逻辑需要在产品层面进行详细设计，确保各种边界场景都能正确处理。

## 四、数据权限配置

### 4.1 权限类型定义

根据平台规范，数据权限控制是劳务公司端和工厂端的核心能力，合同管理模块同样需要遵循数据权限配置规则。数据权限控制确保不同角色的人员只能查看和操作其权限范围内的合同数据，既保障了数据安全，又避免了信息泄露。

平台定义了五种数据权限类型，分别是：全部数据（ALL）权限，持有该权限的用户可以查看所有合同数据，无论合同属于哪个部门或由谁创建；本部门数据（DEPARTMENT）权限，持有该权限的用户只能查看本部门创建的合同数据；本部门及以下（DEPARTMENT_AND_BELOW）权限，持有该权限的用户可以查看本部门及下属子部门创建的合同数据；本人数据（SELF）权限，持有该权限的用户只能查看自己创建或负责的合同数据；自定义（CUSTOM）权限，持有该权限的用户可以查看指定部门或人员的合同数据。

### 4.2 权限应用场景

在合同管理模块中，数据权限的具体应用场景如下：

**劳务公司管理员账户**：通常配置为“全部数据“权限，可以查看和操作本公司所有员工创建的合同。这是考虑到管理员需要对公司整体合同情况进行监控和管理。

**劳务公司正式员工账户**：根据其岗位配置的数据权限，可能是“本部门数据”、“本人数据”或“自定义”权限。例如，负责招聘的员工只能查看自己创建的合同，而部门经理可以查看整个招聘部门的合同。

**工厂管理员账户**：由于工厂端主要使用在职管理、离职管理等功能，合同管理模块在工厂端的可见性需要根据业务需求确定。如果工厂需要查看与其相关的合同，可以配置为查看关联合同（即与该工厂工人相关的合同）。

### 4.3 权限配置实现

数据权限的实现需要在API层面进行统一处理。每当用户请求合同列表数据时，系统需要根据用户的权限配置自动添加数据过滤条件。过滤条件的构建需要考虑以下因素：

如果配置了“本人数据“权限，API需要在查询条件中添加“创建人ID等于当前用户ID”的过滤条件；如果配置了“本部门数据“权限，API需要在查询条件中添加“创建人所属部门ID等于当前用户所属部门ID”的过滤条件；如果配置了“本部门及以下“权限，API需要查询当前用户所属部门及其所有子部门的ID列表，然后在查询条件中添加“创建人所属部门ID在指定部门列表中”的过滤条件。

同时，合同详情查询、合同编辑、合同删除等操作也都需要校验当前用户是否具有该合同的访问权限。如果用户无权限访问某合同，API应返回403禁止访问错误。

## 五、与其他模块的关联关系

### 5.1 与工人管理的关联

合同管理与工人管理模块之间存在紧密的数据关联。工人管理模块存储了所有工人的基本信息，包括个人资料、技能证书、工作履历等。合同管理模块中的“乙方“字段需要从工人管理模块中选择人员，这要求两个模块之间建立正确的数据关联。

具体关联逻辑如下：在合同表单中选择乙方时，系统应弹出人员选择器，展示符合条件（通常为已完成面试流程、状态为“面试邀约“或”工厂面试“）的工人列表供选择。选中工人后，系统将工人的基本信息（如姓名、手机号、所属劳务公司等）关联到合同记录中。当合同状态变更为“已签订“时，系统需要自动更新该工人的生命周期状态为”进厂”。

此外，工人详情页面可以展示该工人关联的合同信息，包括合同编号、签订日期、合同状态、合同金额等。这方便了劳务公司员工在一个页面中查看工人的完整信息。

### 5.2 与在职管理的关联

合同管理与在职管理模块的关联主要体现在结算依据和状态联动两个方面。在职管理模块中的工资管理、考勤管理等功能需要参考合同中约定的结算方式和工资金额。例如，月结工人每月按照合同约定金额计算工资，日结工人则按照实际工作天数和日薪计算。

当工人处于“在职“状态时，其合同信息应该能够被在职管理模块正确引用。具体实现方式可以是：在职管理模块在计算工资时，通过工人ID查询其当前有效的合同记录，获取结算方式和合同金额等关键信息。

同时，合同状态变更可能触发在职管理模块的相关处理。例如，如果合同被取消，系统应通知在职管理模块及时处理工人的后续事宜。

### 5.3 与离职管理的关联

合同管理与离职管理模块的关联主要体现在合同终止处理方面。当工人提交离职申请或被标记为离职时，系统需要检查该工人是否有未到期的有效合同。如果有，系统应提示相关人员处理合同终止事宜。

离职管理模块可以在办理离职手续时展示工人的合同信息，包括合同编号、签订日期、到期日期、合同金额等，供操作员参考。完成离职办理后，对应的合同状态应自动变更为“已取消”，并记录变更原因为“工人离职”。

### 5.4 与结算管理的关联

合同管理与结算管理模块的关联主要体现在结算依据的提供方面。结算管理模块负责计算和发放工人工资、劳务公司佣金等，需要以合同中约定的结算方式、工资标准等为依据。

具体数据流转如下：合同签订时约定的结算方式（月结、日结、周结）传输到结算管理模块，作为工资计算方式的依据；合同中约定的工资金额（或日薪/周薪标准）作为工资计算的基准；合同状态为“已签订“的工人才会进入结算流程，未签订合同的工人不会被纳入结算名单。

结算管理模块可能还需要展示合同的相关信息，如在工资条中显示合同编号、结算方式等，方便工人核对。

### 5.5 与系统管理的关联

合同管理与系统管理模块的关联主要体现在以下方面：合同状态、结算方式等字典数据由系统管理模块的“字典管理“功能维护；打印模板由系统管理模块的“打印管理“功能配置；操作权限由系统管理模块的“菜单配置“和”角色权限“功能控制。

这些关联关系确保了合同管理模块能够与平台其他模块保持数据一致性和功能协调性。

## 六、UI/UX设计要求

### 6.1 菜单与导航设计

根据ui-design-spec.md的设计规范，合同管理作为一级菜单显示在顶部导航栏中，使用Document图标标识。合同管理下仅包含一个二级菜单“签订合同”，该菜单项对应合同列表页面。二级菜单使用左侧侧边栏显示，支持折叠和展开操作。

页面导航采用四级导航栏结构，即：顶部一级菜单（合同管理）→左侧二级菜单（签订合同）→页面标签栏（当前页面）。不需要在内容区域重复显示页面标题。

进入合同详情页面时，会在页面标签栏新增一个标签；点击标签可以快速切换到对应详情页；详情页支持右键菜单操作，包括刷新、关闭当前页、关闭其他页等。

### 6.2 列表页面设计

合同列表页面采用标准列表页布局，从上到下依次为：查询条件区域（合同状态、结算方式、签订日期范围）、功能按钮栏（新增合同）、数据表格、分页控件。

查询条件区域的表单字段横向排列，使用el-form的inline模式，每个筛选条件使用el-select或el-date-picker组件。查询条件和功能按钮之间使用CommonTable组件的toolbar-left和toolbar-right插槽进行布局。

数据表格使用CommonTable组件，展示字段包括：合同编号、甲方、乙方、结算方式、合同状态、合同签订日期、合同生效日期、合同过期日期、合同金额。表格支持列宽调整、字段排序、文本截断和悬浮提示。合同状态字段使用el-tag组件展示，根据状态显示不同颜色（未签订-灰、签订中-橙、已签订-绿、已取消-红）。

操作列包含查看、编辑、删除三个操作按钮。编辑和删除按钮根据合同状态动态显示，只在“未签订“或”签订中“状态时可见。

### 6.3 表单页面设计

合同表单页面采用两列布局，左侧为主要信息区，右侧为合同内容编辑区。表单使用CommonForm组件构建，字段标签宽度统一设置为140px。

甲方字段默认带出当前登录账户所属企业名称，操作员可以手动修改。乙方字段使用自定义组件PersonSelect，点击后弹出人员选择对话框，从工人列表中选择。结算方式使用下拉选择器，提供月结、日结、周结三个选项。合同签订日期、生效日期、过期日期使用日期选择器，支持日期格式的显示和存储。

合同内容字段使用RichTextEditor富文本编辑器组件，支持输入合同正文内容，支持加粗、斜体、标题等常用格式，支持图片插入。编辑器高度设置为400px，placeholder提示文字为“请输入合同内容”。

合同附件字段使用FileUpload组件，支持拖拽上传和多文件上传，单个文件大小限制50MB，最多上传10个文件。附件列表显示文件名、大小和上传时间，支持预览和下载。

表单底部居中悬浮放置操作按钮，包括“保存草稿“和”提交“两个按钮。保存草稿将数据保存为草稿状态，提交则进行表单验证后提交到服务器。

### 6.4 详情页面设计

合同详情页面采用卡片式布局，从上到下依次展示：基本信息卡片、合同内容卡片、合同附件卡片、操作记录卡片。各卡片之间使用20px的间距，卡片内部使用el-descriptions组件展示键值对信息。

基本信息卡片包含合同编号、甲方、乙方、结算方式、合同状态、签订日期、生效日期、过期日期、金额等字段。合同状态使用el-tag组件展示，带有状态颜色。合同金额使用特殊样式突出显示，使用红色字体和较大字号。

合同内容卡片直接渲染富文本HTML内容，使用浅灰色背景区分。附件卡片展示附件列表，每行显示附件图标、文件名、大小、预览和下载按钮。操作记录卡片使用el-timeline组件展示时间线形式的历史操作记录。

详情页面底部居中悬浮放置操作按钮，根据合同状态动态显示。对于“未签订“或”签订中“状态的合同，显示编辑、删除按钮；对于所有状态的合同，显示打印按钮。按钮之间保持12px的间距，使用居中对齐。

### 6.5 组件交互规范

根据项目规范的组件设计原则，合同管理模块需要遵循以下交互规范：

按钮操作规范方面，新增和编辑操作采用跳转方式，点击按钮后跳转到对应的表单页面进行操作；删除操作采用弹窗确认方式，点击删除按钮后弹出确认对话框，进行二次确认后再执行删除；其他需要确认的操作同样采用弹窗确认方式。

日期时间组件规范方面，所有日期选择器需要使用中文界面，包括月份、星期、按钮文字等；日期格式统一显示为“YYYY-MM-DD”形式；placeholder提示文字使用中文，如“选择日期”。

表单验证规范方面，必填字段使用红色星号标识；验证失败时在字段下方显示错误提示文字；提交前进行整体表单验证，确保数据完整性。

响应式设计规范方面，页面需要适配桌面端和平板端；小屏幕设备上表单从两列布局切换为单列布局；功能按钮栏在小屏幕上调整排列方向。

## 七、审批流程配置

### 7.1 审核状态定义

合同管理模块的审批流程采用标准的三状态审核模型，包括：未审核、审核中、已审核（包括审核通过和审核驳回两种结果）。这三个状态与合同状态（未签订、签订中、已签订、已取消）相互独立但有关联。

具体对应关系如下：合同刚创建时，合同状态为“未签订”，审核状态为“未审核”；合同提交审核后，合同状态保持“未签订”，审核状态变更为“审核中”；审核通过后，合同状态可变更“签订中”或直接“已签订”，审核状态变更为“已审核（通过）”；审核驳回后，合同状态保持“未签订”，审核状态变更为“已审核（驳回）”，并记录驳回原因。

### 7.2 审批流程配置方案

根据平台规范，审批流程配置采用可插拔的设计，即合同审批流程可以配置为启用或禁用。具体配置方案如下：

**流程启用场景**：如果业务要求合同需要经过审批才能生效，则在系统管理模块的“流程管理“中启用合同审批流程。审批流程可以配置为单级审批或多级审批，每级审批可以指定审批人和审批方式（会签或或签）。

**流程禁用场景**：如果业务要求合同无需审批，创建后直接生效，则可以禁用合同审批流程。禁用后，合同创建后状态直接为“未签订”，操作员可以将状态变更为“签订中“或”已签订”，无需经过审核步骤。

### 7.3 审批记录管理

无论审批流程是否启用，合同管理模块都需要完整记录审批相关的操作历史。审批记录包括以下内容：

**提交记录**：记录合同提交审核的时间、提交人、提交备注等信息。

**审核记录**：记录每次审核的审核节点、审核人、审核时间、审核结果（通过/驳回）、审核意见。多次审核时，审核记录按时间倒序展示。

**状态变更记录**：记录合同状态变更的时间、操作人、变更前状态、变更后状态、变更原因。状态变更记录是操作记录的子集，专门用于跟踪合同状态的流转。

这些记录存储在操作记录卡片中，方便用户追溯合同的处理历史，也便于审计和合规检查。

### 7.4 待办任务集成

当合同提交审核后，系统应自动生成对应的待办任务，推送给审批人。待办任务显示在审批人的工作台“待办任务“列表中。审批人点击待办任务可以跳转到合同详情页面进行审核处理。

审核处理完成后，待办任务状态变更为“已办”，从待办列表中移除并进入已办列表。同时，系统可以发送消息通知给合同提交人，告知审核结果。

待办任务的具体实现由工作中心模块的“待办任务“功能提供，合同管理模块只需要在适当的业务节点调用接口创建待办任务即可。

## 九、接口需求

### 9.1 接口设计原则

根据平台接口设计规范，合同管理模块的API接口需要遵循以下原则：URL使用小写字母和连字符，采用RESTful风格；版本号放在URL中，如/api/v1；统一响应格式，包含code、message、data、timestamp字段；支持分页、排序、筛选等标准查询参数；所有接口应用数据权限过滤。

### 9.2 主要接口清单

以下是合同管理模块需要实现的主要接口：

**合同列表查询接口**：GET /api/v1/contracts。请求参数包括：page（页码）、pageSize（每页条数）、sortField（排序字段）、sortOrder（排序方式 asc/desc）、keyword（关键词搜索）、status（合同状态筛选）、settlementMethod（结算方式筛选）、signDateStart/signDateEnd（签订日期范围筛选）。响应数据包括分页的合同列表和总数。

**合同详情查询接口**：GET /api/v1/contracts/{id}。返回指定合同的完整信息，包括基本信息、合同内容、附件列表、操作记录等。

**合同创建接口**：POST /api/v1/contracts。请求体包含合同的所有字段信息。创建成功后返回新创建合同的ID，状态默认为“未签订”，审核状态默认为“未审核”（如果启用了审批流程）。

**合同更新接口**：PUT /api/v1/contracts/{id}。请求体包含需要更新的字段信息。更新成功后返回更新后的合同信息。此接口仅允许对“未签订“或”签订中“状态的合同进行操作。

**合同删除接口**：DELETE /api/v1/contracts/{id}。删除指定ID的合同。此接口仅允许对“未签订“或”签订中“状态的合同进行操作。

**合同状态变更接口**：PATCH /api/v1/contracts/{id}/status。请求体包含目标状态和变更原因。状态变更需要校验是否符合状态流转规则。

**合同提交审核接口**：POST /api/v1/contracts/{id}/submit。提交合同进入审核流程。此接口触发待办任务的创建。

**合同审核接口**：POST /api/v1/contracts/{id}/approve。请求体包含审核结果（通过/驳回）和审核意见。审核通过后更新审核状态和合同状态。

**合同附件上传接口**：POST /api/v1/contracts/{id}/attachments。请求体为MultipartFile文件流。返回上传成功的附件信息。

**合同附件删除接口**：DELETE /api/v1/contracts/{id}/attachments/{attachmentId}。删除指定附件。

**工人选择接口**：GET /api/v1/workers/select。返回符合合同签订条件的工人列表，供人员选择器调用。筛选条件包括：工人状态为“面试邀约“或”工厂面试”、所属劳务公司与当前用户一致等。

### 9.3 响应格式规范

所有接口统一使用以下响应格式：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    // 业务数据
  },
  "timestamp": 1704067200000
}
```

分页响应格式：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "list": [
      // 数据列表
    ],
    "total": 100,
    "page": 1,
    "pageSize": 10
  },
  "timestamp": 1704067200000
}
```

错误响应格式：

```json
{
  "code": 400,
  "message": "请求参数错误",
  "data": null,
  "timestamp": 1704067200000
}
```
