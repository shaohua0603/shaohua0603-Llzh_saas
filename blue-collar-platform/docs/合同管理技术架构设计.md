## 一、概述

### 1.1 文档目的

本文档为蓝领智汇平台合同管理模块的技术架构设计文档，旨在明确模块的前后端技术架构、数据库设计、接口规范以及与其他模块的集成方案。本文档基于《合同管理需求分析报告》和《蓝领智汇平台开发规范》编写，为开发团队提供详细的技术指引，确保开发过程的技术一致性和系统可维护性。

合同管理模块是蓝领智汇平台的核心业务模块之一，主要服务于劳务公司与工人之间的劳动合同管理。该模块需要支持合同的创建、编辑、查看、删除、状态变更等全生命周期管理功能，同时需要与工人管理、在职管理、离职管理、结算管理等模块进行数据交互，实现业务闭环。

### 1.2 设计原则

本模块的技术架构设计遵循以下核心原则：

首先是标准化原则，所有接口遵循RESTful设计规范，遵循平台统一的命名规范和编码规范，确保系统的一致性和可维护性。

其次是模块化原则，采用分层架构设计，将表示层、业务层、数据层清晰分离，降低模块间的耦合度，提高代码复用性。

第三是安全性原则，实施完善的用户认证和授权机制，实现细粒度的数据权限控制，保障业务数据的安全性。

第四是性能优先原则，合理设计数据库索引和缓存策略，优化API响应时间，确保系统在高并发场景下的稳定运行。

第五是扩展性原则，设计时充分考虑业务扩展需求，预留必要的扩展接口，支持未来功能的平滑迭代。

## 二、系统架构总览

### 2.1 整体架构

合同管理模块采用典型的三层架构设计，包括前端展示层、业务逻辑层和数据访问层。前端采用Vue 3 + TypeScript + Element Plus技术栈，通过HTTP请求与后端服务进行交互。后端采用RESTful API设计规范，提供标准化的接口服务。数据库采用MySQL关系型数据库，通过数据访问层实现业务数据的持久化存储。

```
┌─────────────────────────────────────────────────────────────────┐
│                        前端展示层                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Vue 3 + TypeScript + Element Plus + CommonTable       │   │
│  │  CommonForm + PersonSelect + RichTextEditor            │   │
│  └─────────────────────────────────────────────────────────┘   │
└──────────────────────────┬──────────────────────────────────────┘
                           │ HTTP / HTTPS
┌──────────────────────────▼──────────────────────────────────────┐
│                        业务逻辑层                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  合同管理API  +  审批执行引擎  +  数据权限服务           │   │
│  └─────────────────────────────────────────────────────────┘   │
└──────────────────────────┬──────────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────────┐
│                        数据访问层                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  MySQL数据库  +  附件存储服务  +  缓存服务               │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 2.3 模块交互关系

合同管理模块与平台其他模块存在密切的交互关系，主要包括以下方面：

与工人管理模块的交互体现在合同中乙方人员的选择，以及合同签订后工人生命周期状态的联动更新。与审批执行引擎的交互体现在合同提交审核时触发审批流程，审批结果影响合同状态变更。与数据权限服务的交互体现在所有合同数据查询都需要经过权限过滤，确保用户只能访问权限范围内的数据。与附件管理服务的交互体现在合同附件的上传、存储和预览功能。与待办任务服务的交互体现在合同提交审核后生成待办任务，推送给审批人处理。

### 3.2 组件结构

#### 3.2.1 页面组件

合同管理模块包含三个核心页面组件，每个组件承担独立的职责：

**合同列表页（Contract.vue）** 负责展示合同数据列表，支持多条件筛选、全局搜索、排序、分页等功能。该页面使用CommonTable组件作为核心展示组件，通过toolbar-left插槽放置查询条件表单，通过toolbar-right插槽放置功能按钮。列表支持批量选择和批量操作，导出功能通过选中数据后调用导出接口实现。

**合同表单页（ContractForm.vue）** 负责合同的创建和编辑功能。该页面使用CommonForm组件构建表单，集成PersonSelect组件实现乙方人员选择，集成RichTextEditor组件实现合同内容的富文本编辑，集成FileUpload组件实现合同附件上传。表单页面支持自动保存草稿功能，间隔30秒自动保存一次，防止数据丢失。

**合同详情页（ContractDetail.vue）** 负责展示合同的完整信息，包括基本信息、合同内容、合同附件和操作记录。页面采用卡片式布局，使用el-descriptions组件展示键值对信息，使用el-timeline组件展示操作记录时间线。底部悬浮放置操作按钮，根据合同状态动态显示编辑、删除、打印等操作。

## 四、后端API设计

### 4.1 接口设计原则

合同管理模块的API接口设计遵循以下原则：

RESTful规范方面，采用标准HTTP方法表示操作类型，GET用于查询，POST用于创建，PUT用于更新，PATCH用于部分更新，DELETE用于删除。URL使用小写字母和连字符，采用名词复数形式表示资源集合，如/contracts表示合同集合。

版本控制方面，API版本号放在URL路径中，如/api/v1/contracts，便于版本管理和向后兼容。

响应格式方面，所有接口采用统一的响应格式，包含code（状态码）、message（消息）、data（数据）和timestamp（时间戳）四个字段。

分页查询方面，支持page（页码）、pageSize（每页条数）、sortField（排序字段）、sortOrder（排序方式）四个分页参数。

数据权限方面，所有查询接口自动应用数据权限过滤，根据当前用户的角色和数据权限配置返回相应的数据。

## 五、数据库设计

### 5.1 数据库设计原则

合同管理模块的数据库设计遵循以下原则：

首先是命名规范方面，表名使用小写字母和下划线，如contracts；字段名使用小写字母和下划线，如contract_no；索引名使用idx_前缀，外键使用fk_前缀。

其次是通用字段方面，所有业务表都包含id（主键）、created_by（创建人）、created_at（创建时间）、updated_by（更新人）、updated_at（更新时间）、deleted_at（删除时间）、remark（备注）七个通用字段。

第三是数据权限方面，所有需要数据权限控制的表都包含creator_id（创建人ID）、department_id（所属部门ID）两个字段，用于数据权限过滤。

第四是软删除方面，采用deleted_at字段进行软删除，查询时默认过滤已删除数据。

## 七、与其他模块的集成方案

### 7.1 与工人管理模块的集成

#### 7.1.1 集成场景

合同管理模块与工人管理模块存在以下集成场景：

乙方选择方面，在合同表单中选择乙方时，需要从工人列表中选择符合条件的工人。可选工人需要满足以下条件：工人状态为“面试邀约”或“工厂面试”或“在职”；工人所属劳务公司与当前用户所属劳务公司一致；工人尚未签订有效合同。

状态联动方面，当合同状态变更为“已签订”时，系统需要自动更新对应工人的生命周期状态为“进厂”。这一联动通过在合同状态变更服务中调用工人状态更新接口实现。

#### 7.1.2 接口对接

工人选择接口调用示例：

```typescript
// 获取可选工人列表
GET /api/v1/workers/selectable?status=INTERVIEW_INVITED,Factory_INTERVIEW,ON_DUTY
```

响应数据中的Worker对象包含id、name、phone、idCard、status、currentCompany等字段，用于在人员选择器中展示。

状态联动实现示例：

```typescript
// 合同状态变更为已签订时，调用工人状态更新接口
async function updateWorkerStatus(workerId: string, newStatus: string) {
  await workerApi.updateStatus(workerId, {
    status: newStatus,
    relatedContractId: contractId,
    updateReason: '合同已签订'
  })
}
```

### 7.2 与审批执行引擎的集成

#### 7.2.1 集成场景

合同审批流程集成包括以下场景：

提交审核方面，当用户提交合同时，如果系统配置了合同审批流程，则触发审批执行引擎创建审批任务。审批任务根据配置的审批节点，生成对应的待办任务推送给审批人。

审核处理方面，审批人在待办任务中对合同进行审核，审核结果通过审批执行引擎回调更新合同状态。

审批记录方面，合同详情页面需要展示完整的审批记录，包括提交记录、审核记录、状态变更记录。

#### 7.2.2 接口对接

审批执行引擎相关接口已在平台中定义，合同模块需要调用以下接口：

```typescript
// 提交审核
import { submitApproval } from '@/api/approvalExecutionApi'

// 提交合同审核
const result = await submitApproval({
  businessType: 'CONTRACT',
  businessId: contractId,
  flowCode: 'CONTRACT_APPROVAL',
  submitterId: currentUserId,
  submitterName: currentUserName,
  submitComment: '提交合同审核'

// 审核合同
import { approve, reject } from '@/api/approvalExecutionApi'

// 审核通过
await approve({
  taskId: todoId,
  businessType: 'CONTRACT',
  businessId: contractId,
  approvalResult: 'approved',
  approvalComment: '合同内容合规，同意签订'
})

// 审核驳回
await reject({
  taskId: todoId,
  businessType: 'CONTRACT',
  businessId: contractId,
  approvalResult: 'rejected',
  approvalComment:驳回原因'
})
```

### 7.3 与数据权限服务的集成

#### 7.3.1 集成场景

数据权限服务在以下场景发挥作用：

列表查询过滤方面，所有合同列表查询都需要经过数据权限过滤，根据当前用户的角色和数据权限配置返回相应的数据。数据权限类型包括全部数据、本部门数据、本部门及以下、本人数据、自定义部门数据。

详情访问控制方面，合同详情查询需要校验当前用户是否有权访问该合同，无权限时返回403错误。

操作权限控制方面，合同编辑、删除等操作需要校验当前用户是否有权操作该合同。

#### 7.3.2 接口对接

数据权限过滤在服务端实现，客户端无需特殊处理。但客户端需要传递当前用户信息用于权限判断：

```typescript
// API请求时自动携带用户Token，服务端解析Token获取用户信息进行权限判断
// 请求头中包含 Authorization: Bearer {token}
```

客户端可以通过以下方式获取当前用户权限信息：

```typescript
import { useUserStore } from '@/stores/user'

const userStore = useUserStore()
const dataPermission = userStore.userInfo?.dataPermission
const departmentId = userStore.userInfo?.departmentId
```

### 7.4 与附件管理服务的集成

#### 7.4.1 集成场景

附件管理服务用于处理合同附件的上传、存储、预览和下载。

文件上传方面，合同附件采用分片上传机制，支持大文件上传，支持断点续传。上传前需要获取上传令牌，防止重复上传。

文件预览方面，支持图片、PDF等常用格式的在线预览。对于不支持预览的格式，提供下载链接。

文件删除方面，删除合同时同步删除关联的附件，或者在附件表中标记为已删除。

#### 7.4.2 接口对接

附件相关接口：

import { getAttachments } from '@/api/attachmentApi'

const attachments = await getAttachments({

### 7.5 与待办任务服务的集成

#### 7.5.1 集成场景

待办任务服务用于处理合同审核的待办事项。

待办创建方面，合同提交审核后，审批执行引擎会调用待办任务服务创建待办任务，推送给审批人。

待办处理方面，审批人处理完审核后，待办任务状态变更为已办。

## 八、技术规范和开发标准

### 8.1 编码规范

#### 8.1.1 命名规范

文件命名方面，Vue组件文件使用大驼峰命名法，如Contract.vue、ContractForm.vue。TypeScript类型定义文件使用小驼峰命名法，如contract.ts。API接口文件使用小驼峰命名法，如contractApi.ts。

变量命名方面，普通变量使用小驼峰命名法，如contractList、currentPage。组件props使用小驼峰命名法。常量使用全大写字母和下划线，如MAX_FILE_SIZE。

函数命名方面，普通函数使用小驼峰命名法，如getContractList、handleSearch。事件处理函数使用handle前缀，如handleAdd、handleDelete。业务逻辑函数使用动词或动词短语，如fetchContract、updateStatus。

#### 8.1.2 代码格式

缩进方面，使用2个空格进行缩进，不使用Tab字符。每行代码长度不超过120个字符。

引号方面，JavaScript字符串使用单引号或反引号，HTML属性值使用双引号。

注释方面，组件顶部添加组件说明注释，复杂逻辑添加详细注释，函数添加参数和返回值说明注释。注释使用中文编写。

#### 8.1.3 TypeScript规范

类型定义方面，为所有变量添加类型定义，为函数参数和返回值添加类型定义。使用接口定义复杂类型，使用泛型提高代码复用性。

类型检查方面，启用严格类型检查，避免使用any类型，使用unknown类型替代any。

### 8.2 组件开发规范

#### 8.2.1 组件结构

Vue组件使用Composition API和script setup语法。组件结构按照以下顺序组织：template模板、script脚本、style样式。script部分按照以下顺序组织：import导入、类型定义、组件属性、响应式状态、计算属性、watch监听、生命周期钩子、业务方法。

#### 8.2.2 组件通信

父子通信方面，使用props传递数据，使用emit传递事件。props使用defineProps宏定义，emit使用defineEmits宏定义。

跨组件通信方面，对于多层级的组件通信，使用provide/inject机制。对于全局状态管理，使用Pinia Store。

#### 8.2.3 样式管理

使用scoped样式，避免样式污染全局。避免使用!important。合理使用CSS变量。实现响应式设计，适配不同屏幕尺寸。

### 8.3 API调用规范

#### 8.3.1 请求处理

所有API请求使用try-catch包装，捕获异常并给出友好提示。表单提交、删除等关键操作使用loading状态防止重复提交。文件上传使用FormData格式。

#### 8.3.2 响应处理

统一处理响应码，200表示成功，4xx表示客户端错误，5xx表示服务端错误。分页响应数据解构使用list、total、page、pageSize字段。

#### 8.3.3 错误处理

网络错误提示“网络错误，请稍后重试”。业务错误显示服务端返回的错误消息。未知错误提示“系统错误，请联系管理员”。

### 8.4 安全规范

#### 8.4.1 身份认证

所有API请求需要携带有效的认证Token。Token过期时自动跳转登录页面。敏感操作需要二次确认。

#### 8.4.2 数据校验

前端进行表单数据初步校验。后端进行严格的数据校验。文件上传进行文件类型和大小校验。

#### 8.4.3 XSS防护

用户输入的内容进行HTML转义。富文本内容进行白名单过滤。使用Element Plus的安全过滤机制。

## 十、开发建议

### 10.1 开发顺序

建议按照以下顺序进行开发：

第一阶段实现基础功能，包括合同列表展示、新增合同、编辑合同、删除合同、合同详情展示。第二阶段实现附件功能，包括附件上传、附件预览、附件下载。第三阶段实现审批功能，包括提交审核、审核处理、审批记录展示。第四阶段实现高级功能，包括合同模板、合同导出、合同打印。第五阶段实现优化完善，包括性能优化、用户体验优化、bug修复。

### 10.2 测试要点

功能测试方面，测试合同列表的筛选、排序、分页功能。测试合同的新增、编辑、删除功能。测试合同详情的查看、打印功能。测试附件的上传、预览、下载功能。

表单验证测试方面，测试必填字段验证。测试日期逻辑验证。测试金额数值验证。

权限测试方面，测试不同角色的权限控制。测试数据权限过滤。测试越权访问防护。

性能测试方面，测试大数据量列表加载性能。测试大文件上传性能。测试接口响应时间。

### 10.3 注意事项

合同编号生成方面，新增合同时自动生成合同编号，格式为HT{YYYYMM}{随机4位}。

日期验证方面，合同生效日期不能早于签订日期，合同过期日期必须晚于生效日期。

状态限制方面，已签订和已取消的合同不能编辑和删除。审核中的合同不能编辑和删除。

附件限制方面，单个文件大小不超过50MB，最多上传10个文件。

草稿保存方面，表单支持自动保存草稿，间隔30秒。

数据权限方面，根据用户岗位配置的数据权限过滤合同数据。
