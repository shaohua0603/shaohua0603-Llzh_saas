# 岗位管理功能开发计划

## 一、项目概述

### 1.1 项目背景
岗位管理是蓝领智汇SaaS平台系统管理模块的核心功能之一，负责管理系统中各个岗位的基础信息、数据权限和菜单权限配置。

### 1.2 项目目标
- 完成岗位管理功能的完整开发
- 实现岗位信息的增删改查功能
- 实现数据权限配置功能（5种数据权限类型）
- 实现菜单权限配置功能（树状结构，自动勾选子菜单）
- 确保功能符合项目规范和UI设计要求

### 1.3 项目范围
- 岗位列表页面
- 岗位表单页面（新增/编辑）
- 岗位详情页面
- 岗位管理API接口
- 岗位管理Mock数据
- 岗位管理权限控制

## 二、需求分析

### 2.1 业务需求和功能范围

#### 2.1.1 业务背景
岗位管理是系统权限管理的基础，通过岗位来控制用户的操作权限和数据访问范围。岗位管理需要支持：
- 岗位基础信息管理（岗位名称、所属部门）
- 数据权限配置（控制岗位能查看的数据范围）
- 菜单权限配置（控制岗位能访问的功能模块）
- 岗位的增删改查操作

#### 2.1.2 功能范围
1. **基础信息管理**
   - 岗位名称
   - 所属部门
   - 岗位描述
   - 岗位状态（启用/禁用）

2. **数据权限配置**
   - 全部数据（ALL）：查看所有数据
   - 本部门数据（DEPARTMENT）：仅查看本部门数据
   - 本部门及以下（DEPARTMENT_AND_BELOW）：查看本部门及下属部门数据
   - 本人数据（SELF）：仅查看本人创建/负责的数据
   - 自定义（CUSTOM）：自由选择各个部门，查看各个部门向下的权限

3. **菜单权限配置**
   - 树状结构展示菜单
   - 支持勾选到功能按钮层级
   - 勾选父菜单后自动勾选所有子菜单
   - 支持多选

4. **CRUD操作**
   - 新增岗位
   - 编辑岗位
   - 删除岗位
   - 查看岗位详情
   - 查询岗位列表

#### 2.1.3 数据权限应用场景
- **劳务公司管理员账户**：查看本公司所有数据
- **劳务公司正式员工账户**：根据岗位配置的数据权限查看
- **工厂管理员账户**：查看本工厂所有数据
- **工厂正式员工账户**：根据岗位配置的数据权限查看

### 2.4 操作流程和交互逻辑

#### 2.4.1 新增岗位流程
1. 点击"新增"按钮
2. 跳转到新增岗位页面
3. 填写岗位信息：
   - 岗位名称（必填）
   - 所属部门（必填，下拉选择）
   - 数据权限类型（必填，下拉选择）
   - 自定义部门（当选择"自定义"时显示，多选树）
   - 菜单权限（必填，树状结构多选）
   - 岗位描述（选填）
   - 岗位状态（默认启用）
4. 点击"保存"按钮
5. 表单验证
6. 提交数据到后端
7. 保存成功，跳转回列表页

#### 2.4.2 编辑岗位流程
1. 点击列表中的"编辑"按钮
2. 跳转到编辑岗位页面

5. 点击"保存"按钮
6. 表单验证
7. 提交数据到后端
8. 保存成功，跳转回列表页

2. 弹出二次确认对话框
3. 确认删除

### 2.5 权限控制要求

#### 2.5.1 数据权限控制
- **全部数据（ALL）**：可以查看所有数据
- **本部门数据（DEPARTMENT）**：仅能查看本部门的数据
- **本部门及以下（DEPARTMENT_AND_BELOW）**：可以查看本部门及所有下属部门的数据
- **本人数据（SELF）**：仅能查看自己创建/负责的数据（包括创建人、负责人、对接人等）
- **自定义（CUSTOM）**：可以自由选择多个部门，查看这些部门及其下属部门的数据

#### 2.5.2 操作权限控制
- 查看列表：需要"岗位管理-查看"权限
- 新增岗位：需要"岗位管理-新增"权限
- 编辑岗位：需要"岗位管理-编辑"权限
- 删除岗位：需要"岗位管理-删除"权限
- 查看详情：需要"岗位管理-查看"权限
- 导出数据：需要"岗位管理-导出"权限

#### 2.5.3 数据验证权限
- 岗位名称不能重复
- 岗位名称不能为空
- 所属部门不能为空
- 数据权限类型不能为空
- 菜单权限不能为空

### 2.6 与其他模块的关联关系

#### 2.6.1 部门管理模块
- 岗位必须关联到部门
- 数据权限配置需要依赖部门树结构
- 自定义数据权限需要从部门树中选择

#### 2.6.2 正式员工模块
- 正式员工必须关联到岗位
- 员工的数据权限由岗位继承
- 员工的菜单权限由岗位继承

#### 2.6.3 菜单配置模块
- 菜单权限配置需要从菜单配置中获取菜单树
- 菜单权限配置需要支持到功能按钮层级

#### 2.6.4 规则配置模块
- 数据权限规则需要在规则配置中定义
- 数据权限过滤规则需要在规则配置中配置

#### 2.6.5 审批流程模块
- 岗位变更可能需要审批

```java
public class DataPermissionFilter {
    public Specification<Position> buildSpecification(Position position) {
        return (root, query, cb) -> {
            switch (position.getDataPermissionType()) {
                case "ALL":
                    return null; // 查询所有数据
                case "DEPARTMENT":
                    return cb.equal(root.get("departmentId"), position.getDepartmentId());
                case "DEPARTMENT_AND_BELOW":
                    List<String> departmentIds = getDepartmentAndBelow(position.getDepartmentId());
                    return root.get("departmentId").in(departmentIds);
                case "SELF":
                    return cb.or(
                        cb.equal(root.get("createdBy"), position.getCreatedBy()),
                        cb.equal(root.get("manager"), position.getCreatedBy()),
                        cb.equal(root.get("contact"), position.getCreatedBy())
                    );
                case "CUSTOM":
                    List<String> customDepartmentIds = position.getCustomDepartments();
                    List<String> allDepartmentIds = new ArrayList<>();
                    for (String deptId : customDepartmentIds) {
                        allDepartmentIds.addAll(getDepartmentAndBelow(deptId));
                    }
                    return root.get("departmentId").in(allDepartmentIds);
                default:
                    return null;
            }
        };
    }
}
```

```typescript
// directives/permission.ts
import type { Directive, DirectiveBinding } from 'vue';
import { useUserStore } from '@/stores/user';

export const permission: Directive = {
  mounted(el: HTMLElement, binding: DirectiveBinding) {
    const { value } = binding;
    const userStore = useUserStore();
    const permissions = userStore.permissions;

    if (value && value instanceof Array && value.length > 0) {
      const hasPermission = permissions.some(permission => {
        return value.includes(permission);
      });

      if (!hasPermission) {
        el.parentNode?.removeChild(el);
      }
    }
  }
};
```

## 六、风险与应对

### 6.1 技术风险

#### 风险1: 菜单权限树性能问题
- **风险描述**: 菜单权限树节点数量较多时，可能导致渲染性能下降
- **影响程度**: 中
- **应对措施**:
  - 使用虚拟滚动优化长列表渲染
  - 使用懒加载，按需加载子节点
  - 使用缓存机制，避免重复渲染
  - 对菜单树进行分页或分级加载

#### 风险2: 数据权限过滤性能问题
- **风险描述**: 数据权限过滤逻辑复杂，可能导致查询性能下降
- **影响程度**: 高
- **应对措施**:
  - 优化数据库索引
  - 使用缓存机制，缓存权限过滤结果
  - 使用预编译SQL，提高查询效率
  - 对大数据量进行分页处理

#### 风险3: 前后端数据格式不一致
- **风险描述**: 前后端对数据格式的理解不一致，导致数据解析错误
- **影响程度**: 中
- **应对措施**:
  - 使用统一的接口文档（Swagger）
  - 使用TypeScript类型定义，确保类型一致
  - 建立前后端联调机制，及时发现和解决问题
  - 使用Mock数据，提前验证数据格式

### 6.2 进度风险

#### 风险1: 任务依赖导致进度延迟
- **风险描述**: 任务之间存在依赖关系，前置任务延迟会导致后续任务延迟
- **影响程度**: 高
